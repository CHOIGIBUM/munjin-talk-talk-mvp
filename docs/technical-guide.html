문진톡톡 MVP 기술 안내서

내부 구조와 JSON 스키마를 한 번에 보는 설명서

환자 음성 문진이 AWS 백엔드와 React 프론트를 거쳐 의사용 원페이퍼와 환자 안내문으로 바뀌는 과정을 팀원이 바로 이해할 수 있도록 정리했습니다.

Frontend React/Vite 화면, 녹음, API 호출, JSON을 UI로 표시
Backend API Gateway, Lambda, DynamoDB, S3 세션 처리
AWS AI Transcribe 음성 인식, Bedrock Nova 의미 구조화
Output 의사용 원페이퍼, 환자 안내문, TTS와 출력 화면

핵심 문장: 환자 1명의 전체 문진 결과는 DynamoDB의 session JSON 하나에 누적되고, 화면은 그 JSON을 읽어 보여줍니다.

전체 흐름 폴더 구조 프론트엔드 백엔드 API AI 역할 JSON 스키마 호출 순서 배포 디버깅

Overview

1. 전체 흐름

말로 입력한 문진이 어떤 순서로 저장되고 화면에 보이는지 가장 큰 그림입니다.

접수 직원이 환자 확인 후 문진 세션 생성
환자 태블릿 초진/재진 선택 후 Q1~Q4 음성 답변
STT S3 업로드 후 Transcribe가 텍스트 변환
AI 구조화 Bedrock이 증상, 맥락, 질문을 JSON으로 분해
저장 DynamoDB session JSON에 누적 저장
원페이퍼 의사가 증상, 원문, 확인 항목, 질문 확인
안내문 의사 답변을 환자 안내문으로 변환

쉽게 말하면

프론트는 화면과 녹음 담당, 백엔드는 저장과 AI 처리 담당, AWS AI는 음성 인식과 의미 구조화 담당입니다.

가장 중요한 저장 단위

환자 1명은 session 1개입니다. Q1~Q4 답변, 증상 슬롯, 원페이퍼, 안내문이 같은 session_id 아래에 쌓입니다.

현재 MVP 기준

Bedrock 실호출을 사용하며, `ALLOW_RULE_FALLBACK=false`라서 LLM 실패 시 규칙 결과로 몰래 대체하지 않습니다.

Frontend

3. 프론트엔드 화면과 역할

현재 운영 배포에서는 실제 세션이 없으면 환자 태블릿, 원페이퍼, 안내문 메뉴가 비활성화됩니다.

경로 사용자 화면 컴포넌트 역할
/staff 접수 직원 ReceptionView 환자 정보 입력, 문진 세션 생성, 오늘 접수 목록 확인
/patient/{sessionId} 환자 PatientKioskView, PatientFlow 초진/재진 선택, Q1~Q4 음성 문진, STT 확인
/doctor/queue 의료진 DoctorQueueView 문진 완료, 우선 확인 환자 목록 확인
/doctor/{sessionId} 의료진 DoctorView, DoctorOnePager, DoctorAgendaPanel 증상, 원문 quote, 맥락 단서, 질문 확인 및 답변 작성
/guide/{sessionId} 환자/직원 PatientGuideScreen 진료 후 안내문, TTS, 종이 출력

프론트 API 호출 흐름

  • createIntakeSession() → POST /sessions
  • uploadAudio() → POST /upload-url → S3 PUT
  • getTranscript() → GET /transcribe-result 반복 조회
  • processTranscript() → /extract → /match → /validate

조회/진료 후 API

  • getDoctorQueue(): 접수/의사 대기열 조회
  • getOnePager(): 원페이퍼 JSON 조회
  • submitDoctorResponse(): 의사 답변/강조사항 저장
  • getPatientGuide(): 환자 안내문 조회

AI Orchestration

6. LLM/AI 역할 분리

지금 구조는 Q별로 모델을 완전히 따로 둔 것보다는, 작업 난이도별로 Bedrock 모델을 라우팅하는 형태입니다.

단계 모델/서비스 하는 일 JSON 영향
음성 인식 Amazon Transcribe 환자 음성을 한국어 텍스트로 변환 responses.Qx.text
의미 분해 Bedrock Nova Pro Q1/Q3 증상, 경과, 새 증상 의미 단위 추출 spans, clinical_clues
가벼운 구조화 Bedrock Nova Lite 복약, 환자 질문, 안내문 등 상대적으로 가벼운 처리 structured.questions, patient_guide.items
증상 매칭 BM25 + Titan embedding diseases_cleaned.json과 symptom_index.json에서 rule-base로 만든 증상 문서를 검색하고 표준 증상명/점수를 산출 matched_slots, ir_trace
원페이퍼 리뷰 Bedrock Nova Pro 확인 항목과 EMR 문장을 의료진용으로 다듬음 review_items, transfer_text
안전 감지 Lambda rule 객혈/피 표현 등 우선 확인 플래그 safety_flags, risk
/match는 Bedrock이 붙인 slot_ref를 최종 정답으로 쓰지 않습니다. 환자 원문과 표준화 표현을 다시 검색 질의로 만들고, BM25 점수와 Titan embedding 거리, 표준명 유사도를 함께 계산해 matched_slots를 만듭니다. 검색 문서는 symptom_retrieval_dataset.json을 쓰지 않고 두 원천 JSON에서 Lambda cold-start 시 rule-base로 생성합니다.

JSON Schema

7. 내부 JSON 스키마

여기 있는 JSON이 실제 화면의 원천입니다. 특히 responses와 onepager를 보면 대부분의 문제를 추적할 수 있습니다.

7-1. Session JSON 전체

{
  "session_id": "s_1780...",
  "queue_number": 1,
  "created_at": "2026-...",
  "updated_at": "2026-...",
  "status": "waiting_tablet | in_progress | completed | needs_priority | reviewed | staff_help",
  "visit_type": "initial | followup",
  "risk": "none | high",
  "patient": { "...": "..." },
  "audio": { "Q1": { "...": "..." } },
  "responses": { "Q1": { "...": "..." } },
  "question_results": { "...": "responses copy" },
  "onepager": { "...": "..." },
  "doctor_review": { "...": "..." },
  "patient_guide": { "...": "..." },
  "safety_flag": { "...": "..." }
}

읽는 법

  • 세션 1개가 환자 1명의 문진 전체를 대표합니다.
  • 접수 직후에도 빈 onepager 초안이 만들어집니다.
  • 각 문항 확정 때마다 responses.Qx가 채워집니다.
  • Q4까지 완료되면 completed 또는 needs_priority가 됩니다.
  • 의사가 답변을 보내면 doctor_review와 patient_guide가 추가됩니다.

7-2. 환자 정보

"patient": {
  "name": "최*자",
  "full_name": "최영자",
  "birth_date": "1950-09-17",
  "age": 75,
  "gender": "여성",
  "receipt_id": "R-9596",
  "department": "이비인후과",
  "doctor": "이민우",
  "phone": "010-0000-0000"
}

생성 위치

  • 프론트 입력: ReceptionView.jsx
  • 백엔드 생성: create_session()
  • 나이 계산: birth_date 기준
  • 화면 표시는 보통 마스킹된 name 중심입니다.

    "key": "sessions/{session_id}/Q1.webm",
    "content_type": "audio/webm",
    "uploaded_at": "2026-..."

"responses": {

중요한 흐름

  • 브라우저가 S3 presigned URL로 오디오를 직접 업로드합니다.
  • /transcribe-result가 Transcribe 작업을 시작하거나 결과를 조회합니다.
  • 환자가 화면에서 텍스트를 확인하기 전에는 confirmed=false입니다.
